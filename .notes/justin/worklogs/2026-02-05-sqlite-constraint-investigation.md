# Investigate SQLITE_CONSTRAINT in Deterministic Linking 2026-02-05

## Initialized investigation
We are investigating a `SQLITE_CONSTRAINT` error occurring during the `deterministic_linking` phase (Phase 6):
`UNIQUE constraint failed: moments.document_id, moments.micro_paths_hash`

This is unexpected because Phase 6 typically handles linking, not moment insertion.

## Investigated the SQLITE_CONSTRAINT error
We have identified the root cause of the `SQLITE_CONSTRAINT` on `(document_id, micro_paths_hash)`. 

### The Findings
1.  **Brittle ID Generation**: Macro moment IDs (UUIDs) are generated in Phase 5 using the `macroIndex` (the 0-based index in the LLM's response).
2.  **Constraint Collision**: There is a UNIQUE constraint on `(document_id, micro_paths_hash)` in the `moments` table. This ensures that a given set of source chunks (the `micro_paths_hash`) only appears once per document.
3.  **The Trigger**: If a simulation run processes multiple versions of the same document (common in Cursor conversations) or retries a failed Phase 5, and the LLM produces the same macro moment at a **different index**, the system tries to `INSERT` it as a new moment (due to the new ID). SQLite then throws a constraint failure because the `(document_id, micro_paths_hash)` already exists under the old ID.

### Proof of Failure during Phase 6
While the collision could happen in Phase 5, Phase 6 also calls `addMoment` to update the parent links and audit logs. If the DB already contains a row that conflicts with the update, it will throw. 

### Proposed Fix: Content-Addressed IDs
We will shift the macro moment ID generation from **index-based** to **content-addressed** (or at least stable across content shifts).
- If a moment has micro-chunks, we should incorporate the `micro_paths_hash` into its ID instead of the `macroIndex`.
- This ensures that if the same set of chunks is synthesized twice (across retries or versions), it resolves to the same `momentId`, allowing `addMoment` to perform an `UPDATE` instead of a conflicting `INSERT`.

## Implemented Content-Addressed IDs
We have updated the `materialize_moments` orchestrator to use the `micro_paths_hash` as a seed for macro moment IDs. 

1.  **Orchestrator Update**: `computeMaterializedMomentIdentity` now prefers the `micro_paths_hash` over the `macroIndex` when generating IDs.
2.  **Stability**: This ensures that even if the order of moments in the LLM response changes, or if a document is re-processed with minor changes that don't affect a specific moment's chunks, the ID remains stable.
3.  **Conflict Resolution**: This allows `addMoment` to perform an `UPDATE` on existing conceptual moments instead of failing with a `SQLITE_CONSTRAINT` error.

## Drafted PR
### Title
Resolve SQLITE_CONSTRAINT via Content-Addressed Macro Moment IDs

### Description
In our simulation pipeline, macro moments are generated by synthesizing sequences of micro-moments. Previously, the unique identity (`moment.id`) of these macro moments was partially derived from their index in the LLM's response. 

This created a brittle dependency on the LLM's output order. If the same conceptual moment (the same set of source chunks) was produced at a different index—due to retries, document versioning, or LLM variance—it was assigned a new UUID. This triggered a `SQLITE_CONSTRAINT` violation against the unique index on `(document_id, micro_paths_hash)`, as the system would attempt to `INSERT` a "new" moment that conflicted with an existing entry for the same content.

This change shifts macro moment identity generation from index-based to content-addressed. 

We now incorporate the `micro_paths_hash` (the hash of the source micro-chunks) directly into the seed used for generating the moment's UUID. As a result, conceptual duplicates within a document now resolve to the same stable ID. This enables the database layer to perform a safe `UPDATE` (upsert behavior) instead of a conflicting `INSERT`. 

We have maintained the `macroIndex` as a fallback seed for instances where no micro-chunks are present, ensuring stability across all moment types while maintaining backward compatibility for existing simulation states.
