## Investigation Discoveries

### 1. Systemic Artifact Loss (Phase 5 Missing)
We confirmed that `timeline_fit` (Phase 8) failed for issue 552 because it could not find the output of `materialize_moments` (Phase 5). This error pattern (`No materialize_moments output found`) appeared for many other documents in the same simulation run.

### 2. Zombie Ditching Mechanism Triggered
The simulation supervisor correctly identified that `timeline_fit` was failing repeatedly for these documents. After 3 failed attempts, it "ditched" the document (marked it as processed/skipped) to prevent the simulation from stalling. This explains why the issue was never fully processed into the graph.

### 3. Root Cause Hypothesis: Flawed Restart Logic
We traced the issue to `restartSimulationRunFromPhase` in `runs.ts`. When restarting a simulation from a later phase (e.g., `candidate_sets`), the code aggressively clears `processed_phases_json` or associated state for *all* phases, or fails to re-constitute the necessary artifact state for intermediate phases if they are not re-run. Specifically, if a run is restarted but the artifacts from previous phases like `materialize_moments` are missing (perhaps due to a prior cleanup or ephemeral storage issue during a crash), the system has no way to recover them without a full replay.

## 2026-02-11 Log Vacuum Findings
We identified a systematic "Zombie Ditching" issue using the expanded logs.

### 1. The Ditching Pattern
Logs showed that hundreds of documents were being "ditched" in `micro_batches`, `macro_synthesis`, `macro_classification`, and `materialize_moments` phases.
Example: `2026-02-10T20:14:34.311Z [warn] phase.document_ditched {"phase":"materialize_moments","r2Key":"...","attempts":4}`

### 2. Root Cause: Strict Timeouts + Missing Heartbeats
The `SimulationWorker` initializes a `heartbeat` function that updates the `updated_at` field in the database, but this function is **never called** by any phase implementation.
The simulation runner (`runner.ts`) uses a `zombieThreshold` of only **30 seconds**. For large documents that take minutes to process via LLM batches, the runner increments "attempts" every 30s and eventually ditches the document after 4 attempts.

### 3. Hollow Progression
When a document is ditched, it is marked as `processed` for that phase to allow the simulation to advance. This leads to documents reaching later phases like `timeline_fit` without any artifacts from previous phases, resulting in the error: `No materialize_moments output found`.

**Evidence Trace**: Document `4af71ae7-51eb-4ac1-a46e-7283ebb17fb4` was ditched in `micro_batches` at `2026-02-10T20:07:34.999Z` after 4 attempts. Despite this, the simulation advanced it through `macro_synthesis`, `macro_classification`, and `materialize_moments` (ditching it in each), until it finally failed hard in `timeline_fit` at `2026-02-10T20:27:06.565Z`.

## 2026-02-11 Reasoning for Previous (Rejected) Heartbeat Implementation
We implemented manual heartbeats because we observed that the `updated_at` latch (which the supervisor uses to determine if a document is "alive") only updates at the very end of a worker's `executePhase`. In long-running phases like `micro_batches` (with 50+ embeddings) or `macro_synthesis` (heavy synthesis), a single unit of work can exceed 30s. We believed that without manual internal ticking, the supervisor would inevitably kill healthy jobs during their execution.

However, we now recognize that manual internal heartbeats are a "business logic pollutant" and functionally equivalent to `sleep` patterns if not handled by infrastructure.

## RFC: Simplified Zombie Resiliency

### 2000ft View Narrative
The "Zombie Ditching" mechanism is currently too aggressive. It triggers after 30s of database silence, which is often exceeded by legitimate queue wait times and deep LLM processing. We will resolve this by increasing the supervisor's patience and removing the non-functional heartbeat code that has accumulated in the worker.
