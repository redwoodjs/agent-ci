# 2026-01-12-pr-933-issue-552-linking

## Noticed missing linkage between GitHub PR #933 and issue #552

PR #933 explicitly says it closes issue #552, but the knowledge graph does not show them connected.

The first thing I noticed is that PR #933 does not have any macro moments persisted, which means it cannot attach under other timelines.

## Plan for narrowing it down

- Check whether PR #933 was ingested and processed by the synthesis pipeline.
- If it was processed, check whether macro gating dropped all macro moments (or classification failed).
- Surface per-document synthesis audit records in the audit UI for documents that have zero moments, so the reason is visible without needing an existing moment to click.

## Surfaced document synthesis audit logs on the ingestion file page

The Knowledge Graph audit view already renders per-document synthesis audit events, but it only shows them when a moment exists and is selected.

I updated the ingestion file detail page to fetch and display synthesis audit logs for the document id directly. This means documents that produced zero moments can still show:

- macro gating counts (input macro count, output macro count, dropped count)
- macro classification parse failures
- any other synthesis events written to the document audit log

## Hypothesis: 'closes issue' is treated as noise by synthesis and gating

The macro synthesis prompt explicitly tells the model to exclude administrative state changes (including closing an issue) unless the text includes a concrete technical decision or implementation change.

The engine also applies a post-synthesis noise filter that can drop macro moments mentioning 'closed issue' when they do not contain technical signal keywords. If PR #933's macro output is mostly about closing the issue, it might get filtered down to zero macro moments and never persist.

## Found the actual blocker: score threshold gate prevented #552 from being evaluated

After getting a macro moment persisted for PR #933, the smart linker still did not attach it under issue #552.

The smart linker has a hard score threshold (0.75) applied before the timeline-fit check runs. Issue #552 was present as a candidate but scored 0.715, so it was rejected as below-threshold and never reached the timeline-fit check.

This means even if the proposed moment includes an explicit close reference, it can be ignored if the vector score is slightly under the threshold.

## Change: treat explicit issue references as eligible candidates

I updated the smart linker so that when the proposed moment and a candidate share an issue reference token (for example `#552`), and the candidate document id corresponds to that same issue number, the candidate bypasses the score threshold and is considered first.

I also added a guidance line to the timeline-fit prompt to treat explicit close/fix/resolve references as strong anchors, but the main fix is to ensure these candidates make it past the pre-LLM threshold gate.

## Direction: separate candidate generation from timeline fit decision

This incident is a concrete example of why parent selection benefits from treating explicit anchors as a first-class input to candidate generation, rather than relying on a fixed vector threshold.

I captured a proposal in `docs/architecture/timeline-fit-as-linking-plugin.md` to separate:

- candidate generators (vector search, anchor match)
- timeline fit decision (bounded chain context evaluation)
